À propos

Je suis étudiant en 2ème année de BUT Informatique parcours Déploiement d'Applications Communicantes et Sécurisées (DACS) à l'Université de Lille. Mon parcours m'a amené à travailler aussi bien sur le code (développement Java, web) que sur l'infrastructure (Linux, Docker, réseaux).

Après la licence, j'envisage une poursuite en master ou école d'ingénieur avec une spécialisation DevOps / Cloud, ou une intégration directe en entreprise via alternance.

SAÉ B · S2 — Déploiement d'une application
Infrastructure de déploiement multi-VMs
Projet en binôme · 5 semaines · Semestres 2–3
Linux Debian Bash Docker PostgreSQL Traefik SSH Nginx VirtualBox
Contexte & objectifs

L'objectif de cette SAÉ était de déployer une infrastructure complète de communication basée sur le protocole Matrix (serveur Synapse + client Element Web), en partant de zéro sur des machines virtuelles Debian. Le projet couvrait 5 semaines progressives : mise en place SSH, configuration réseau statique, installation PostgreSQL, déploiement de Synapse, mise en place d'un reverse proxy Nginx, puis migration vers une architecture multi-VMs segmentée.

Mon rôle & contributions

J'ai contribué à la totalité des étapes : configuration SSH avec clés asymétriques, scripting Bash pour automatiser les installations, paramétrage PostgreSQL avec contrôle d'accès par IP, et mise en place d'un reverse proxy Traefik en conteneur Docker. J'ai également rédigé un script de déploiement automatisé (deploy.sh) permettant de déployer l'infrastructure complète ou d'ajouter un nouveau client Dolibarr avec une seule commande.

J'ai pris en charge la partie sécurisation : désactivation du login root SSH, configuration de sudo, contrôle des flux réseau entre les VMs.

Compétences mobilisées
Hard skills
Administration Linux Scripting Bash avancé Réseau TCP/IP PostgreSQL Docker Reverse proxy Cryptographie SSH
Soft skills
Documentation rigoureuse Travail en binôme Résolution d'incidents Autonomie
Analyse réflexive

La principale difficulté a été la gestion des problèmes réseau entre VMs, notamment les règles de pare-feu et les permissions PostgreSQL. J'ai appris à diagnostiquer méthodiquement : logs, test de connectivité couche par couche, isolation du problème. Ce projet m'a également appris l'importance d'une documentation au fil de l'eau — refaire une installation sans notes aurait pris trois fois plus de temps. Aujourd'hui, je ferais davantage appel à Docker Compose pour simplifier l'orchestration multi-services.

SAÉ 3.02 · S3 — Développement d'applications
Jeu de Labyrinthes
Projet en équipe de 4 · 10 semaines · Semestre 3 · 2025–2026
Java JavaFX MVC Algorithmique Git / GitLab Agile
Contexte & objectifs

Développement en équipe de 4 d'un jeu de labyrinthes complet en Java avec interface JavaFX. L'objectif pédagogique couvrait le développement efficace (R3.02), l'analyse UML (R3.03) et la qualité de code (R3.04). Le projet s'est déroulé sur 2 jalons : le premier couvrant les labyrinthes aléatoires avec modes libre et progression, le second ajoutant les labyrinthes parfaits et des fonctionnalités originales.

Les labyrinthes sont générés procéduralement avec contraintes : pourcentage de murs, distance minimale entrée-sortie garantie, et plusieurs étapes de difficulté progressive (vision restreinte, carte dynamique…).

Mon rôle & contributions

J'ai pris en charge l'algorithme de génération des labyrinthes parfaits (basé sur la fusion d'ensembles / Kruskal) et la vérification du chemin entrée-sortie via BFS. J'ai également participé à la mise en place de l'architecture MVC stricte et à la coordination Git (gestion des branches, merge requests).

Compétences mobilisées
Hard skills
Java avancé JavaFX / UI Algorithmes de graphes BFS / Kruskal Architecture MVC Tests unitaires JUnit
Soft skills
Travail en équipe Agile Revues de code Gestion des délais Communication
Analyse réflexive

Travailler en équipe de 4 sur une base de code partagée a été la plus grande source d'apprentissage — et de friction. Les conflits Git, les désaccords sur l'architecture, la répartition des tâches : tout cela m'a appris à communiquer précisément sur le code. J'aurais dû établir dès le début des conventions de nommage et une architecture clairement documentée. Ce que je retiens surtout : une bonne architecture prise tôt évite des heures de refactoring tard.

SAÉ · S1–S2 — Infrastructure Forge
Forge logicielle Gitea sous Debian 12
Projet individuel · Semestres 1–2
Linux Debian Gitea Bash Cron SSH
Contexte & objectifs

Installation, sécurisation et maintenance d'une forge logicielle Gitea auto-hébergée sur une VM Debian 12. L'objectif était de mettre en place un environnement de développement collaboratif fonctionnel : dépôts Git, gestion des utilisateurs, webhooks, et sauvegardes automatisées.

Mon rôle & contributions

Installation from scratch de Gitea, configuration du reverse proxy Nginx, mise en place de scripts Bash d'automatisation des sauvegardes via cron, et durcissement de la configuration SSH (désactivation de l'accès root, authentification par clés uniquement).

Compétences mobilisées
Hard skills
Administration système Bash / Cron Nginx config Sécurité SSH Gestion services systemd
Soft skills
Autonomie Démarche documentaire Rigueur
Analyse réflexive

Ce projet m'a initié à l'administration système réelle. La difficulté principale a été la gestion des permissions et des droits sous Linux — une erreur de chmod peut tout casser silencieusement. J'ai appris à tester de manière systématique après chaque modification. Avec le recul, j'aurais utilisé Docker dès le départ pour isoler le service et simplifier les mises à jour.
